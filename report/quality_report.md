# Отчёт о качестве данных

## 1. Введение

**Датасет:** Bike Sharing Demand  
**Источник:** https://www.kaggle.com/competitions/bike-sharing-demand/data  
**Размер:** 10 936 строк × 12 столбцов.  

**Предметная область:** Датасет содержит данные о количестве арендованных велосипедов в зависимости от времени суток, погодных условий и календарных дней (праздники/выходные).  

**Условный заказчик:** Сервис велопроката или департамент городского транспорта.  

**Возможные задачи ИАД:**
1. **Прогнозирование спроса:** Предсказание количества велосипедов (`count`) для оптимизации их распределения по станциям.
2. **Поиск аномалий:** Выявление дней с неестественно низким спросом (возможные сбои системы) или аномально высоким спросом (мероприятия, праздники).
3. **Профилирование поведения:** Анализ влияния погоды на разные категории пользователей.

## 2. Паспорт датасета

| Столбец | Тип | Описание | Примечания |
|---------|-----|----------|------------|
| `datetime` | datetime64 | Дата и время (с точностью до часа) | Ключевой идентификатор |
| `season` | int64 | Сезон (1: весна, 2: лето, 3: осень, 4: зима) | Категориальный, требует кодирования |
| `holiday` | int64 | Праздник (0/1) | Бинарный |
| `workingday` | int64 | Рабочий день (0/1) | Бинарный |
| `weather` | float64 | Погода (1: ясно ... 4: ливень) | **Есть пропуски (~5%)** |
| `temp` | float64 | Температура воздуха | Непрерывный |
| `atemp` | float64 | Ощущаемая температура | Непрерывный |
| `humidity` | float64 | Влажность (%) | **Пропуски (10.2%) + Аномалии (250)** |
| `windspeed` | float64 | Скорость ветра | Есть пропуски (2.9%) |
| `casual` | int64 | Аренда незарегистрированными | Часть таргета |
| `registered` | int64 | Аренда зарегистрированными | Часть таргета |
| `count` | int64 | **Общее число аренд** | **Целевая переменная (Target)** |

## 3. Основные проблемы качества данных

| № | Тип проблемы | Описание | Потенциальный риск | Предлагаемое решение |
|---|--------------|----------|-------------------|---------------------|
| 1 | **Пропуски** | **`humidity`: 1119 (10.23%)**<br>`weather`: 545 (4.98%)<br>`windspeed`: 322 (2.94%) | Потеря 1/10 части данных при удалении строк недопустима. | **Заполнение:**<br>- `humidity`/`windspeed`: медианой.<br>- `weather`: модой. |
| 2 | **Аномальные значения** | `humidity` max = **250.0**. | Физически невозможно (максимум 100%). Искажает статистику. | Значения > 100 заменить на 100 или на медиану. |
| 3 | **Дубликаты** | Найдено **50 полных дубликатов**. | Утечка данных (Data Leakage) и смещение весов модели. | Полное удаление дубликатов (`drop_duplicates`). |
| 4 | **Распределение таргета** | `count`: Mean=191, Median=145, Max=977.<br>**310 выбросов** по IQR. | Модель будет плохо предсказывать пиковые значения. | Логарифмирование целевой переменной. |
| 5 | **Типы признаков** | `season` и `weather` числовые, но по смыслу - категории. | Алгоритмы могут найти ложный порядковый смысл (4 > 1). | Применить One-Hot Encoding. |

## 4. План улучшения качества

**Приоритет 1 (критично):**  
1.  Удалить 50 дублирующихся строк.
2.  Исправить аномалию во влажности (значения 250 привести к норме).
3.  Заполнить пропуски в `humidity` (медиана) и `weather` (мода). 10% пропусков во влажности - это слишком много для удаления.

**Приоритет 2:**  
1.  Распарсить `datetime` на `hour`, `month`, `weekday`. Графики показывают сильную зависимость спроса от времени.
2.  Преобразовать `season` и `weather` в бинарные признаки (OHE).

**Приоритет 3:** Прологарифмировать `count` для уменьшения влияния выбросов и приведения распределения к нормальному.  

## 5. Краткое резюме

Датасет обладает хорошим потенциалом для прогнозирования, но требует очистки. Наличие **10.2% пропусков** в ключевом признаке (влажность) и ошибок ввода (250%) делает невозможным использование данных. После реализации плана улучшению  качество данных станет приемлемым для обучения моделей регрессии.
